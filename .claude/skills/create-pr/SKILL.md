# PR 작성 스킬

코드 변경사항만 나열하는 PR은 리뷰어에게 도움이 안 된다. **왜 이렇게 만들었는지**, **어떤 의사결정을 내렸는지**가 PR의 핵심이다.

## 워크플로우

### 1단계: 변경사항 파악

아래 명령을 **병렬로** 실행하여 변경사항을 분석한다:

```bash
# 커밋 히스토리
git log main..HEAD --oneline

# 코드 diff
git diff main...HEAD --stat
git diff main...HEAD

# 현재 브랜치명
git branch --show-current

# 작업 상태
git status
```

- 브랜치명에서 이슈 번호가 보이면 `gh issue view <번호>`로 이슈 맥락 확인
- 변경된 핵심 파일을 읽어서 구체적으로 이해

### 2단계: 맥락 판단

**현재 대화에서 이미 작업을 함께 진행한 경우** → 맥락을 이미 알고 있으므로 바로 3단계로 진행

**새 세션이거나 맥락이 부족한 경우** → AskUserQuestion으로 질문:

- 이 변경의 목적/의도는?
- 대안이 있었다면 왜 이 방식을 선택했는지?
- 리뷰어가 특히 봐야 할 부분이 있는지?

### 3단계: PR 작성 및 생성

**PR 제목** 형식: `{type}({scope}): {설명}` (기존 커밋 컨벤션 따름)

**PR 본문** 템플릿:

```markdown
## 무엇을, 왜

{이 PR의 목적과 의도. 코드 변경이 아니라 해결하려는 문제를 설명}

## 주요 변경

- {변경 내용 + 그렇게 한 이유}
- {변경 내용 + 그렇게 한 이유}

## 의사결정

{대안 대비 선택한 방식의 근거, 트레이드오프. 없으면 섹션 생략}
```

작성 후 **사용자에게 최종 확인**을 받고 `gh pr create`를 실행한다.

## 규칙

- PR 본문은 **한글**로 작성
- "파일 X를 수정했습니다" 같은 변경사항 나열 금지. **왜** 그렇게 했는지를 쓴다
- 의사결정 섹션은 실제 트레이드오프가 있을 때만 포함
- 이슈가 연결되면 본문에 `Closes #번호` 포함
